# TemichevVet - threat model

Дата: 2026-06-03

Этот документ нужен, чтобы любой разработчик понимал, какие обходы нельзя допускать в боте.

## 1. Что защищаем

- `BOT_TOKEN` Telegram-бота.
- `OPENAI_API_KEY` и совместимые LLM-ключи.
- YooKassa `SHOP_ID` и `SECRET_KEY`.
- `ADMIN_IDS` и доступ к Telegram-админке.
- `bot.db` / Docker-папку `data/` с пользователями, питомцами, историей и платежами.
- Права пользователей на своих питомцев, напоминания, историю, наблюдения и подписку Plus.
- GitHub-репозиторий и доступы разработчиков.
- VPS, systemd/Docker-конфигурацию и бэкапы.

## 2. Границы доверия

- Telegram-пользователь не доверенный: любой текст, callback и `pet_id` может быть подделан.
- Callback-кнопки не являются авторизацией. Каждый callback должен заново проверять текущего Telegram-пользователя и владельца данных.
- `.env` доверенный только на сервере/локальной машине владельца. В git он не попадает.
- YooKassa доверенная только после запроса статуса платежа с серверной стороны.
- GitHub доверенный только для кода без секретов. Ключи и база не передаются через issues, PR, коммиты и скриншоты.

## 3. Основные риски и защита

### Утечка токена Telegram

Риск: чужой человек получает `BOT_TOKEN`, запускает второго бота, читает сообщения или ломает UX.

Защита:

- `.env` в `.gitignore`;
- `make security-check` ищет токены в git-visible файлах;
- токены не писать в issues, PR, README, логи и отчёты;
- при утечке сразу перевыпустить токен через BotFather и перезапустить сервис.

### Обход админки

Риск: пользователь вводит `админ` / `admin` или нажимает поддельный `admin:*` callback.

Защита:

- админка доступна только ID из `ADMIN_IDS`;
- `ADMIN_CHAT_ID` не даёт доступ, он только для уведомлений;
- в `production` без явного `ADMIN_IDS` selftest должен остановить запуск;
- неавторизованные попытки логируются и уведомляют владельца с cooldown.
- админские действия и отказанные попытки пишутся в `admin_audit_log`.

### Спам и перегрузка сценариев

Риск: пользователь или бот-скрипт часто вызывает `/start`, обратную связь, проверки оплаты или вход в разбор жалобы.

Защита:

- `RateLimitMiddleware` ограничивает частые сообщения и callback-и на пользователя;
- администраторы из `ADMIN_IDS` освобождены от лимита для служебных проверок;
- лимиты in-memory и действуют в рамках одного процесса. Для нескольких процессов на VPS нужен внешний лимитер или один активный процесс на токен.

### Подделка `pet_id`

Риск: пользователь подставляет чужой ID питомца в callback и пытается открыть карточку, историю, напоминания, вес или вакцинации.

Защита:

- перед показом или изменением данных проверять `owner_id == current_user.id`;
- использовать `get_pet_for_user(owner_id, pet_id)` для новых обработчиков;
- не считать callback-кнопку доказательством владения питомцем.

### Подделка оплаты Plus

Риск: пользователь нажимает `Я оплатил` без оплаты, подставляет чужой `payment_id` или меняет сумму/план.

Защита:

- Plus выдаётся только после серверной проверки YooKassa;
- проверяются `status=succeeded`, `paid=true`, `currency=RUB`, сумма, `metadata.plan_code`, `metadata.user_id`, `metadata.telegram_id`;
- ручная выдача Plus обычному пользователю запрещена;
- live-платёж проверять только через VPS/тестовую YooKassa-конфигурацию.

### Утечка базы

Риск: `bot.db`, `data/`, логи или архивы попадают в git или отправляются стороннему разработчику.

Защита:

- `bot.db`, `data/`, `*.log` игнорируются git;
- бэкапы хранить отдельно от репозитория;
- перед передачей проекта запускать `git status --short` и `make security-check`;
- дампы базы перед анализом обезличивать, если они уходят внешнему разработчику.

### Компрометация GitHub

Риск: разработчик получает лишние права, пушит код с backdoor, случайно добавляет workflow или секреты.

Защита:

- минимальные права: read/write для веток, admin только владельцу;
- изменения через PR;
- включить branch protection для `main`;
- включить GitHub secret scanning, если доступно;
- не хранить production `.env` в GitHub Secrets без необходимости.

### Компрометация VPS

Риск: на сервере остаются старые токены, слабые права на `.env`, нет бэкапа или работает два процесса с одним токеном.

Защита:

- `.env` читать только владельцу процесса;
- перед обновлением делать бэкап кода, `.env` и базы;
- проверять, что запущен один процесс на один Telegram-токен;
- после смены токена перезапускать сервис и проверять `/start`.

## 4. Правила для новых функций

- Если функция принимает `user_id`, `telegram_id`, `pet_id`, `reminder_id`, `payment_id`, она должна проверять владельца.
- Если функция меняет подписку, она должна иметь отдельную проверку платежа или админ-доступа.
- Если функция показывает аналитику, CSV или пользователей, она должна быть доступна только `ADMIN_IDS`.
- Если функция добавляет новый секрет в `.env`, нужно обновить `.env.example`, README, release checklist и security checklist.
- Если функция добавляет внешний сервис, нужно описать, какие ключи нужны и как их отозвать.

## 5. Команды проверки

```bash
make check
make security-check
git diff --check
git status --short
```

`make security-check` не печатает найденные секреты целиком. Он показывает только файл, строку и тип риска.
